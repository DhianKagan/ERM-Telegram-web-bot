# AGENTS.md — инструкции для ассистента (ERM-Telegram-web-bot, v3.2)

Обновлено: 22 Mar 2026
Timezone: Europe/Kyiv

## 0) Роль файла

Этот файл — короткая operational-версия правил для работы в репозитории.
Подробности и полный baseline — в `.openai/assistant_instructions.json`.

Приоритет источников:

1. `.openai/assistant_instructions.json`
2. `AGENTS.md`
3. `README.md`
4. `docs/*.md`
5. остальное

Если найден дублирующий или устаревший промпт/документ, это нужно явно отметить и предложить убрать/обновить, чтобы не было «двух правд».

## 1) Формат начала ответа

Первая строка каждого ответа:

`Mode: <Lite|Standard|Research+>; Timezone: Europe/Kyiv`

Практическое правило: задачи с кодом, git, проверками и PR почти всегда `Standard`.

## 2) Даты и актуальность

- Любые относительные даты дублировать абсолютной датой `DD MMM YYYY` по timezone `Europe/Kyiv`.
- Любые факты после 2024 года проверять через веб, если они чувствительны ко времени: релизы, API, цены, правила, текущие статусы, `latest/current/today/recent`.
- Если веб недоступен, явно писать: `Риск устаревания: веб отключён.`

## 3) U/C policy

- Если критерии успеха, ограничения или ожидаемый результат неясны, считать `U > 0`.
- При `U > 0` задать **ровно 1–2 уточняющих вопроса** и остановиться.
- Не продолжать «наугад», если это может привести к неверным или лишним изменениям.

## 4) Правила работы с репозиторием

- **Read before write**: сначала читать релевантные файлы.
- **Minimal changes**: менять только необходимое.
- **Config over code**: если можно решить конфигом, предпочесть конфиг.
- **Be explicit**: если что-то не проверено — так и написать.
- Не выводить секреты, токены, ключи, содержимое `.env`, PII.
- Без явной необходимости не изменять:
  - `apps/api/src/generated/**`
  - `packages/types/generated/**`
  - `docs/archive/**`

## 5) Когда менять файлы

- Если запрос только на анализ/ревью/план — не менять файлы без явного запроса.
- Если пользователь прямо просит улучшить, исправить или обновить файл — можно вносить минимальные изменения сразу после чтения релевантного контекста.
- Не делать побочные рефакторинги.

## 6) Верификация

Если файл(ы) изменены, выполнить применимые проверки и явно перечислить:

- что запускалось;
- что не запускалось;
- почему.

Базовый список:

```bash
pnpm -w -s install
pnpm --filter shared build
pnpm -r --filter '!shared' build
pnpm --filter apps/api exec tsc -b
pnpm -w test
pnpm -w lint
```

Опционально:

```bash
docker build -t ermbot:test -f Dockerfile .
```

## 7) Git / PR

- Ветка: `fix/<short-description>` или `feat/<short-description>`.
- Commit: `<type>(<scope>): short summary`.
- В PR обязательно указать:
  - что сделано и почему;
  - DoD-чеклист (`build` / `tsc` / `tests` / `lint`);
  - как проверить;
  - нужные выдержки логов.
- В заголовке PR обязательно указать модель, например: `Codex GPT-5.2`.

## 8) Формат результата

- Если изменений файлов нет — не генерировать патчи и полные файлы.
- Если изменения файлов есть — по умолчанию отдавать полные файлы в формате:

```text
--- File: path/to/file.ext
<полное содержимое файла>
--- End File
```

- Unified diff добавлять, если пользователь явно попросил `diff` или это реально упрощает ревью.

## 9) Критерий качества инструкций

Инструкции должны быть:

- непротиворечивыми;
- короткими, но достаточными;
- проверяемыми;
- без повторов JSON/system prompt без необходимости;
- без требований, конфликтующих со средой выполнения.
