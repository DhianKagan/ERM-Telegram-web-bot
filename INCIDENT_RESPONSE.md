# План реагирования на инциденты

## Когда запускать runbook

Запускайте этот путь при любом из событий:

- подозрение на компрометацию `BOT_TOKEN`, `JWT_SECRET`, `SESSION_SECRET`, Railway / GitHub secrets;
- публичная утечка секрета в коммите, логах, артефактах или issue;
- повторяющиеся `401/403` аномалии, указывающие на сломанный auth / CSRF-flow;
- high / critical alert по supply-chain, если подтверждён затронутый runtime;
- несанкционированный доступ к admin-функциям, ролям или данным пользователей.

## Владелец и эскалация

- **Incident commander:** maintainer on duty.
- **Технический владелец auth / cookies / CSRF:** API owner.
- **Технический владелец CSP / клиентского поведения:** Web owner.
- **Release / deploy owner:** maintainer, выполняющий выкладку и ротацию секретов.

Если владелец недоступен, incident commander назначает временного заместителя в incident ticket.

## Response path

1. **Containment.** Отозвать скомпрометированный токен/секрет, остановить опасную интеграцию, при необходимости временно отключить affected workflow или доступ.
2. **Rotation.** Перевыпустить секреты в BotFather / GitHub Secrets / Railway variables и убедиться, что старые значения более не принимаются.
3. **Deploy.** Перезапустить все затронутые инстансы бота, API и worker-процессы.
4. **Validation.** Проверить auth flow, health-check, критичные admin-операции и логи на повторные обращения со старым секретом.
5. **Communication.** Оповестить команду и, если нужно, пользователей о возможных сбоях и окне восстановления.
6. **Recovery hardening.** Восстановить роли/доступы, пересмотреть allowlist/headers/limits и зафиксировать временные компенсирующие меры.
7. **Closure.** Создать postmortem и приложить evidence of completion.

## Evidence of completion

Инцидент можно закрыть только когда есть все артефакты:

- ссылка на incident ticket / security advisory;
- запись о том, какие секреты были отозваны и когда;
- подтверждение успешного deploy / restart после ротации;
- выдержки логов или метрик, показывающие возврат сервиса в норму;
- postmortem с причиной, impact и follow-up задачами.

## Базовый чек-лист

1. **Отозвать токен бота** через BotFather и сгенерировать новый.
2. **Обновить переменные** окружения в Railway / GitHub Secrets и локальных защищённых хранилищах.
3. **Перезапустить** все развернутые инстансы бота, API и worker.
4. **Проверить логи** на предмет подозрительной активности и повторных обращений со старым секретом.
5. **Проверить авторизацию**: `/health`, `/api/v1/csrf`, логин admin UI, refresh flow, критичные маршруты.
6. **Восстановить роли** при необходимости: убедиться, что поле `access` у администраторов равно `2`.
7. **Провести стресс-тест** по инструкции `docs/stress_plan.md`, если инцидент затрагивал доступность/нагрузку.
